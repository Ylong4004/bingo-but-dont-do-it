# DDI 新词条扩展计划（待审查，不在本轮实现）

> 方案日期：2026-07-16
> 状态：仅设计与排期；本轮成品不新增这些词条、不改变现有 171 条词池。
> 审查重点：公平性、可检测性、性能、语音隐私，以及每批词条的具体文案。

## 1. 先改“规则模型”，再扩数量

当前实现基本是“一种物品/方块对应一个 `DDITriggerType`”。继续按这个方式扩展，会产生数百个枚举和重复检测分支，也很难做到数据包或模组物品兼容。建议先把检测层拆成稳定信号，再让词条用参数描述匹配条件：

```text
游戏事件 / 采样器
  -> DDISignal（放置、破坏、合成、拾取、棋盘得分、移动、语音……）
  -> RuleDefinition（信号类型 + 参数谓词 + 进度/截止时间）
  -> 当前 objective 的规则求值器
  -> DDIObjectiveManager 统一扣血、记历史、换词和同步
```

建议的规则字段：

- `id`：唯一词条 ID；
- `repeatKey`：玩法意义上的去重键，同队触发后本局不再抽取；
- `displayKey`：语言键，不再把中文文案写入服务端逻辑；
- `signal`：例如 `ITEM_CRAFTED`、`BLOCK_STOOD_ON`、`BINGO_TILE_CAPTURED`；
- `predicate`：物品 ID、方块标签、目标玩家、棋盘坐标等参数；
- `progress`：次数、累计距离、持续时间或截止条件；
- `availability`：缺少所需模组、目标或棋盘条件时，从抽取池排除；
- `weight/tier`：后续用于常见度、难度和比赛阶段配重。

数据规则建议从资源 JSON 加载，代码只实现信号和通用求值器。第一阶段仍只允许内置且经过验证的规则，之后再考虑数据包扩展。

## 2. 第一批：扩充现有物品与方块类

目标不是机械覆盖注册表中的每个对象，而是覆盖正常 Bingo 前中期经常接触、检测语义明确的对象。按小批次提交，每批都附带“成功路径、取消/失败路径、同 tick 去重”测试。

### 2.1 合成结果

优先加入：木板、木棍、箱子、熔炉、火把、床、船、面包、桶、盾牌、剪刀，以及常用材质的镐/斧/剑/铲/锄和基础护甲。

- 以最终从合成结果槽取走物品为准；只摆好配方但没有取走不能触发。
- 配方一次产出多个物品仍算一次动作，避免一次合成多次扣血。
- 同类材质可用物品标签或参数集合，不为每个木材颜色新增检测代码。

### 2.2 丢弃与拾取

优先加入：原木/木板、圆石/深板岩圆石、泥土、沙子、沙砾、煤炭、铁/铜/金、钻石、红石、食物、基础工具和 Bingo 常见目标物。

- 丢弃继续以原版 `DROPPED` 统计或成功生成丢弃物为准，覆盖 Q、Ctrl+Q 和背包 THROW。
- 拾取以物品真正进入背包的数量变化为准；背包已满、事件取消不触发。
- “丢弃任意方块”和“丢弃泥土”等变体共享清晰的 `repeatKey` 分组规则，避免文字不同但实际规则完全相同的重复抽取。

### 2.3 站立、破坏、放置与持有

优先覆盖草方块、泥土、石头、圆石、深板岩、沙子、沙砾、常用原木/木板/树叶、常见矿石、工作台、熔炉和箱子。

- 站立使用脚下碰撞面与方块标签，而不是只比较显示名称。
- 放置只在操作成功后触发；打开容器、使用物品或放置失败不能误算。
- 对木材、树叶、羊毛等有大量变体的对象使用原版标签；只在玩法确实需要时区分具体颜色/树种。
- 危险或极少出现的对象不进入默认池，可放入后续“困难词池”。

## 3. 第二批：与 Bingo 棋盘结合

需要在 Bingo 的“格子真正被判定完成”事务出口增加稳定事件，例如：

```text
BingoTileCaptured(gameId, revision, teamKey, playerId, row, column, objectiveId, tier)
```

DDI 只消费已经被 Bingo 接受的得分事件，不能自行扫描背包推测格子完成。候选词条如下。

### 3.1 直接触发类

- 在当前词条倒计时结束前完成任意一个格子；
- 完成指定格子，例如 `(2,2)`；坐标文案统一使用玩家可见的 1～5 行/列；
- 完成中心、四角、边缘、指定行、指定列或任一对角线上的格子；
- 完成某个难度/类别的格子；
- 在一个词条周期内完成 2 个或 3 个格子；
- 完成使队伍形成一条线、达到指定总格数或暂时取得领先的格子；
- 队友完成格子（个人玩法）或指定队员完成格子；
- 在另一队刚得分后的短时间内立刻得分。

### 3.2 截止失败类

- 倒计时结束前没有完成任何格子；
- 倒计时结束前没有完成指定区域的格子；
- 一段时间内队伍总格数没有增加；
- 队伍只差一格形成连线但在期限内没有完成。

这类规则不能复用当前“到时无惩罚、只换词”的逻辑。规则需要明确 `ON_ACTION` 与 `ON_DEADLINE_MISSED` 两种结算方式：到期时先由当前规则判断是否扣血，再统一换词。

### 3.3 公平性限制

- 指定坐标必须在当前棋盘有效，已被本队完成的格子不能再作为“完成此格”词条；无候选时不抽取。
- 隐藏词不能要求玩家完成一个已被规则锁死或客观不可达的目标。
- 共享/锁定棋盘模式下要分别定义“完成”“占领”“重复提交”的含义。
- 所有事件携带 `gameId + revision`，旧局或换词后的迟到事件不能扣血。
- “领先/追平”等全局状态词条先放实验池，避免不同 Bingo 模式下语义不一致。

## 4. 第三批：移动与玩家交互

候选词条：

- 步行/疾跑/游泳/乘船累计移动 25、50、100 格；
- 连续向某个方向移动，或在时间窗内离出生点/队友达到指定距离；
- 攻击指定玩家、攻击任意敌队玩家、被队友/敌队玩家攻击；
- 对指定玩家造成一定累计伤害，或在一定时间内互相攻击；
- 与村民、门、床、按钮、拉杆、特定容器或实体成功交互；
- 与队友交换物品、拾取队友刚丢出的物品；
- 进入另一维度、返回原维度，或在两个棋盘得分之间移动指定距离。

实现约束：

- 距离按相邻 tick 的实际路径段累计，传送、死亡重生、换维度和异常大位移不计入；不同移动方式分开统计。
- “特定玩家”在分配时固定 UUID，并显示可识别名称；目标离线、淘汰或不再有效时立即无惩罚重抽。
- 伤害必须使用服务端最终接受的伤害结果，同一次攻击只产生一个 action token。
- 只维护当前词条需要的计数器，不能为所有玩家无条件运行所有距离和交互扫描。

## 5. 第四批：语音关键词（独立可选实验功能）

仓库已有 `integration-voicechat` 和 Simple Voice Chat 分组能力。语音词条建议在这个集成边界上增加独立桥接，但把 ASR、本地模型和同意机制做成可关闭组件；依赖或模型不可用时，所有语音规则从词池排除。

Simple Voice Chat 官方 API 提供服务端 `MicrophonePacketEvent`，可取得玩家连接和 Opus 数据；官方列出的 [Voice Chat Interaction 示例项目](https://modrepo.de/minecraft/voicechat/api/example_projects) 也使用这一事件处理麦克风包。接入应依据 [官方 API 入门](https://modrepo.de/minecraft/voicechat/api/getting_started)、[事件注册说明](https://modrepo.de/minecraft/voicechat/api/registering_events)、[`MicrophonePacketEvent` Javadoc](https://voicechat.modrepo.de/de/maxhenkel/voicechat/api/events/MicrophonePacketEvent.html) 和 [`MicrophonePacket` Javadoc](https://voicechat.modrepo.de/de/maxhenkel/voicechat/api/packets/MicrophonePacket.html)，不要通过网络抓包绕过语音模组 API。

推荐数据流：

```text
MicrophonePacketEvent
  -> 校验对局、玩家同意状态和当前语音词条
  -> 有界的每玩家 Opus 队列
  -> 每玩家串行解码为 PCM
  -> VAD/静音分句
  -> 本地离线 ASR
  -> 规范化文本与当前关键词/别名匹配
  -> server.execute { 再核对 gameId、objectiveId、assignmentToken 后提交 DDISignal }
```

首版建议评估 [Vosk 官方项目](https://github.com/alphacep/vosk-api)：它有 Java 绑定、离线流式识别和可重配词表，适合只识别当前有效的有限关键词。后续可把 [whisper.cpp](https://github.com/ggml-org/whisper.cpp) 作为高精度可选后端；其官方项目支持 CPU/多种硬件后端和 VAD，但原生库、模型分发、内存与延迟成本更高。

上线前的硬性条件：

- 默认关闭，管理员显式开启，开局向所有玩家提示“语音会在本机/服务器本地即时识别”；
- 玩家可拒绝；拒绝者不能抽到或参与语音词条；
- 不调用云端 ASR，原始 Opus、PCM 和完整转写不落盘、不写普通日志；
- 只处理当前确实需要语音检测的队伍成员，其他音频包立即丢弃；
- ASR 在线程池运行，队列有上限；积压时丢弃旧音频，绝不能阻塞服务器 tick；
- 只用最终识别结果触发，设置置信度、别名、简繁/大小写归一化和冷却；
- 识别返回时必须复核分配令牌，避免旧语句命中新词；
- 承认外放、代说和录音回放无法被可靠区分，并把它写入比赛规则。

## 6. 实施批次与审查门

1. **规则引擎原型**：参数化信号、可用性、进度与截止结算；先迁移 5～10 条现有词验证兼容性。
2. **常用对象 A 包**：合成/拾取/丢弃各约 15～25 条，提交明确清单供审查。
3. **常用方块 B 包**：站立/破坏/放置/持有，优先标签化对象。
4. **Bingo 事件包**：先实现“任意格、无格、中心/四角、指定坐标”，运行稳定后再加领先/连线等状态词。
5. **移动交互包**：先做累计距离和任意敌队玩家，再评估指定目标与交换物品。
6. **语音实验包**：单独开关、单独性能预算、单独隐私验收，不与普通词条批次捆绑发布。

每一批合入前必须提供：词条 ID/中文文案/触发语义表、不可用时的处理、成功与反例测试、8/32 玩家性能结果，以及“同队触发后不再抽取”的自动化验证。
