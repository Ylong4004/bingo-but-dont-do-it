# DDI 调试与语音识别诊断

## 本轮语音修复

现场状态已经证明 Simple Voice Chat 服务、中文模型目录和模型加载流程可以到达
READY，Opus 解码和 16 kHz PCM 也都正常。最终根因是 Windows 中文环境下 JNA
默认按 GBK 传递字符串，而 Vosk 的 C 接口规定 grammar 和 JSON 结果必须使用
UTF-8。于是“钻石”“等一下”等所有中文 grammar 在原生边界损坏，只能识别成
`[unk]`，诊断中就表现为每次结段都增加“空结果”。

当前调整如下：

- 完整短语仍必须与当前词条的受限语法精确匹配。
- 在加载模型及创建识别器前强制 Vosk/JNA 使用 UTF-8；中文 grammar 和结果不再乱码。
- 平均置信度改为 0.55，最低单词置信度改为 0.30。
- 原生结果不含逐词置信度时，只允许“最终文本完整精确匹配”回退；部分结果仍不接受。
- 按玩家隔离的流水线诊断额外显示 PCM 峰值、平均幅度和 Vosk 编码，仍不保存
  音频、识别文本或自定义关键词。

第二个根因是 Vosk 中文小模型的动态语法只接受模型词典中的词元，不会自动给
任意短语分词。例如词典有“帮”和“我”，却没有“帮我”；有“烈焰”和“人”，
却没有“烈焰人”。旧实现把整个短语作为单个词元传入，因此这些词会直接落到
`[unk]`。现在所有内置语音词条别名和局内自定义关键词都经过同一个公共生成器：
始终优先加入原始完整词和全句逐汉字路径；不超过 7 个汉字的连续片段还会生成
全部受限分词。每个口述短语最多 64 种、整份动态语法最多 256 种，并采用轮询
分配，保证别名较多时每个目标也都先得到逐字路径。

这里不把 `yin bu yin` 这类拉丁字符串直接传给中文模型，因为它们不是该模型的
可用词元。Vosk 会把“阴 不 阴”等汉字词元映射到内部声学音节后再与声音比较，
所以这条逐字 grammar 本身就是读音匹配路径；并不要求模型先听写出完整词
“阴不阴”。同音读法在声学层可能命中同一目标，这是按读音触发的预期行为。
实测同一模型下，完整词元“阴不阴”返回 `[unk]`，逐字受限语法返回“阴 不 阴”，
三个词元置信度均为 1.0。该逻辑没有按词条 ID 写特例，局内新增的自定义词同样
会被包装为 `voice:<词语>` 并走这条路径。

“啊啊啊”“哈哈哈”一类纯重复语气词另有很窄的重复次数容错，因为小模型可能
合并或重复这些音节；普通关键词仍要求所有汉字完整匹配。离线白噪声 SNR 10 dB
测试中，新的组合语法仍可识别所测短语，因此当前不额外添加静音填充或噪声门，
避免制造重复结段或截掉较轻的首音节。

## 推荐的局内排查流程

1. 大厅中开启“不要做”和“语音关键词”，玩家执行：

       /bingoprefs ddi_voice_consent true

2. 若要保证下一局抽到测试词条，在大厅执行：

       /bingo ddi word next voice_diamond

   该命令会让下一局所有 DDI 目标首先获得“说出钻石”。队伍共享模式下，
   每个队伍仍然只拥有一份共享词条。

3. 开局后先清空旧计数：

       /bingo ddi voice debug reset

4. 查询测试玩家的资格链：

       /bingo ddi voice debug 玩家名

   应确认：参赛=true、有效=true、授权=true、语音连接=true、
   语音词条=true、目标已发布=true。

5. 玩家说“钻石”并松开 PTT，或停止说话至少约 0.7 秒，再次执行上一步命令。
   命令会直接给出计数、最后阶段和自动判断。

6. 若需要先排除 ASR 之后的逻辑，执行：

       /bingo ddi voice simulate 玩家名

   如果该命令能正常扣血，说明 DDI 服务端结算正常，问题位于音频包、Opus、
   Vosk 或桥接校验阶段。

## 诊断链解释

| 停止位置 | 含义 |
| --- | --- |
| 麦克风包=0 | 服务端没有收到该玩家的 Simple Voice Chat 麦克风事件 |
| 麦克风包增加、命中目标包=0 | 当前玩家没有发布中的语音词条目标 |
| 命中目标包增加、入队=0 | 模型、后端或目标版本门拦截 |
| 入队增加、解码帧=0 | Opus 解码器或工作线程失败 |
| 解码帧增加、结段=0 | 还没有停顿/松开 PTT，且连续语音未达到八秒强制边界 |
| 结段增加、空结果增加 | Vosk 没有得到可用最终词语；先确认 Vosk编码=UTF-8 且 PCM 峰值明显大于 128 |
| 文本不符增加 | Vosk 最终结果不是当前受限关键词 |
| 低置信增加 | 文本正确，但仍低于 0.55/0.30 阈值 |
| 匹配增加、桥接交付不增加 | 分配版本、两秒冷却或会话令牌拒绝 |
| 桥接交付增加、DDI 结算不增加 | Minecraft 主线程尚未处理，或权威规则复检拒绝 |
| DDI 结算增加 | 完整链路成功并已结算扣血 |

## 词条调试命令

所有命令均要求 DDI 管理员权限，默认只允许 OP。

| 命令 | 作用 |
| --- | --- |
| /bingo ddi word next 词条ID | 仅在大厅指定下一局所有目标的首词条；开局后一次性消费 |
| /bingo ddi word next status | 查看下一局首词条 |
| /bingo ddi word next clear | 清除下一局首词条 |
| /bingo ddi word set 玩家 词条ID | 局内为该玩家所属目标强制发放指定词条 |
| /bingo ddi word reroll 玩家 | 局内无惩罚重抽；继续遵守本队已触发词条不再抽取 |
| /bingo ddi word reroll all | 为所有存活目标无惩罚重抽 |
| /bingo ddi word info 词条ID | 查看显示文本、分类、触发类型和规则摘要 |
| /bingo ddi voice debug | 查看全局流水线计数 |
| /bingo ddi voice debug 玩家 | 查看指定玩家的资格链和独立计数 |
| /bingo ddi voice debug reset | 清空全局与各玩家诊断计数 |
| /bingo ddi voice simulate 玩家 | 跳过音频/Vosk，直接测试桥接后的 DDI 扣血逻辑 |

word set 是明确的管理员调试覆盖，因此允许故意再次指定已触发词条；
正常随机抽取和 word reroll 仍严格遵守本队触发历史。
